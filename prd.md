# 产品需求文档 (PRD)：交易信号桌面端监控台 (v1.0)

## 1. 产品概述
本产品是一款轻量级的桌面端交易信号监控工具。通过对接指定的服务器 API，轮询获取加密货币的各项技术指标（如 vegas、divMacd 等）触发信号。产品以极简的悬浮窗/托盘形式常驻桌面，并在第一时间将带有看多/看空属性的信号推送给用户，辅助用户进行交易决策。

## 2. API 接口定义规范 (后端依赖)

### 2.1 鉴权方式
* **机制:** API Key 鉴权。客户端需在请求头（Header）中携带 `x-api-key`。
* **错误响应:** 缺少或错误的 `x-api-key` 时，服务端返回 HTTP `401`。

### 2.2 获取信号接口
* **Endpoint:** `POST /api/open/watch-list/symbol-signals`
* **请求头:**
  * `x-api-key`: String, 系统级 Open Signal API Key
* **请求体 (JSON):**
  * `symbols`: String, 必填，交易对列表，如 `BTCUSDT,ETHUSDT`
  * `periods`: String, 可选，时间周期列表，如 `15m,1h,4D`
  * `signalTypes`: String, 可选，信号类型列表，如 `divMacd,vegas`
  * `page`: Number, 可选，分页页码，默认 `1`
  * `pageSize`: Number, 可选，每页数量，默认 `20`，最大 `100`
* **核心响应字段:**
  * `total`: Number, 总记录数
  * `page`: Number, 当前页码
  * `pageSize`: Number, 每页数量
  * `data`: Array, 信号数据列表
    * `symbol`: String, 交易品种
    * `period`: String, 周期
    * `t`: Number, 该条记录时间戳（保留原始字段）
  * `signals`: Object，包含各策略的触发详情。
    * `sd`: Integer, Side 方向 (`1` = 看多, `-1` = 看空)
    * `t`: Long, 信号触发的时间戳 (毫秒)
    * `read`: Boolean, 是否已读；`false` 表示未读。

### 2.3 设置已读状态接口
* **Endpoint:** `POST /api/open/watch-list/symbol-alert/read-status`
* **请求头:**
  * `x-api-key`: String, 系统级 Open Signal API Key
* **请求体 (JSON):**
  * `symbol`: String, 交易品种，如 `BTCUSDT`
  * `period`: String, 周期，如 `15m`
  * `signalType`: String, 信号类型，如 `divMacd`
  * `read`: Boolean, 是否已读；`true` 标记为已读，`false` 恢复为未读
* **响应:**
  * `true`: 更新成功
  * `false`: 目标记录或信号不存在

### 2.4 删除信号接口
* **Endpoint:** `POST /api/open/watch-list/symbol-alert/delete-signal`
* **请求头:**
  * `x-api-key`: String, 系统级 Open Signal API Key
* **请求体 (JSON):**
  * `symbol`: String, 交易品种，如 `BTCUSDT`
  * `period`: String, 周期，如 `15m`
  * `signalType`: String, 要删除的信号类型，如 `divMacd`
* **响应:**
  * `true`: 删除成功
  * `false`: 目标记录或信号不存在

## 3. 功能需求详细说明 (用户故事拆解)

### 3.1 首次启动与配置
* **授权登录:** 首次打开软件，提供极简的输入框供用户配置 API Key。
* **监控项设置:** 用户可配置关注的 `symbols`、`periods`（如 10D, W, 4D, 3D...15, 10, 5, 1 等）以及具体需要图形化展示的 `signalTypes`（如 Vegas）。

### 3.2 桌面端交互与显示
* **常驻与隐藏:** 支持隐藏到系统托盘，或以半透明悬浮窗形式贴边隐藏在屏幕边缘。
* **置顶显示:** 提供“窗口置顶”开关，确保盯盘时不会被其他应用遮挡。
* **新消息提醒:**
  * 当轮询接口发现任意信号对象的 `read == false` 时，隐藏状态下的窗口自动展开。
  * 触发系统级通知弹窗（可前往设置关闭，支持声音提醒）。
  * UI 界面对新信号增加“未读红点”或高亮呼吸灯效果。
* **状态消除:** 用户在界面上点击或查阅信号后，客户端**立刻在本地 UI 上消除未读状态（乐观更新）**，并在后台静默调用 [2.3 已读接口] 同步给服务器。如果网络失败，则恢复未读状态并提示。

### 3.3 图形化信号模板展示 (核心视图)
* **设计目标:** 直观展示某个具体配置信号（如 vegas）在近期的触发频率与方向。
* **渲染逻辑:** * 界面呈现 **最近 60 根 K 线** 的位置（以 60 个竖条/方块组成的横向时间轴表示）。
  * **前端计算推导:** 通过公式 `(当前时间戳 - 信号触发时间 t) / 当前选择周期的时间跨度`，计算出该信号属于近 60 根 K 线中的哪一根。
  * **视觉区分:** 属于该信号触发的那个竖条，根据方向着色。例如：`sd: 1` 填充绿色（看多），`sd: -1` 填充红色（看空）。未触发信号的竖条显示为暗色/灰色背景。如果计算结果超出了 60 的范围，则该信号不在此模板中渲染。

### 3.4 数据同步与限流保护
* **轮询机制:** 默认采用每 **1 分钟** 一次的后台定时轮询。用户可在设置中自定义轮询频率。
* **限流保护机制:** 客户端强制最低轮询间隔不得低于 10 秒。若接口返回 HTTP 429 (限流) 或 5xx 错误，客户端需自动开启退避策略（例如暂停轮询 3 分钟），并在托盘图标处显示异常标识，防止服务雪崩。

## 4. 非功能性与技术建议

* **技术栈推荐 (Tauri + 前端):** 考虑到你需要开发一款占用极小、常驻后台的桌面程序，且有处理复杂 API 数据和时间计算的需求，推荐使用 **Tauri (Rust 提供底层系统交互 + TypeScript/Node.js 生态构建前端)**。这种组合既能保证编译后体积小巧、内存开销低，又能完美适配 macOS 与 Windows 环境，前端部分的渲染也会非常灵活。

请针对以下问题进行确认，确认后这份 PRD 即可定稿并进入开发阶段：

1. **时区与 K 线切割对齐:** 由于前端需要用 `(当前时间 - 信号时间) / 周期` 来推算 K 线位置，如果是 `1D` (日线) 这种大级别周期，服务端的 K 线是以 UTC 0 点，还是北京时间早上 8 点作为切割点的？（前端需要这个基准值来对齐当前时间的计算，否则可能会出现竖条位置偏移）。

答: 按世界统一时间UTC+0

2. **“60根K线”展示的单一性:** 文档 3.3 中提到，用户可以配置“某个具体（如 vegas）信号”进行展示。在这个 60 根竖条的模板下，用户是**只能选一个**特定信号来看，还是允许**同时选多个**（比如同时看 vegas 和 divMacd）？如果允许多个，同一个竖线上同时出现做多和做空信号时，UI 该如何展示（比如竖条一分为二，或者显示混合色）？

答: 用户可以在界面配置多个组, 一个组是[symbol, signal]等配置

3. **接口一返回的数据结构确认:** `POST /api/open/watch-list/symbol-signals` 接口在同一周期、同一币种下，返回的 `signals` 对象中，每种策略是否**只返回最新的一次**触发时间 `t`？如果它只返回最新的一次，那么 60 根 K 线的视图中，永远最多只会有一个带颜色的竖条被点亮，这样符合你的业务预期吗？

答: 是的, 结果只会有最新的警报, 旧的不存数据的, 也在此接口不返回

   `POST /api/open/watch-list/symbol-alert/read-status` 需要前端传什么参数？是传一个信号的唯一 ID，还是传 `symbol` + `period` + `signalType` 的组合键？

答: `symbol` + `period` + `signalType` 组合
